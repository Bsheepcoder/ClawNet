# ClawNet 关系宪章

> **关系必须建立在安全平等的基础之上**

## 核心原则

### 1. 平等原则 ⚖️
**所有节点生而平等，无中心权威**

- 每个节点都是独立的个体
- 没有节点比其他节点更"高级"
- 节点类型（人/Bot/设备/组织/系统）只表示功能差异，不表示地位高低
- 关系类型和权限级别是功能性的，不是等级性的

### 2. 自愿原则 🤝
**关系必须双方同意，不可强加**

- 任何关系的建立都需要目标节点的明确同意
- 单方面发起的关系请求处于"待确认"状态
- 目标节点有权接受、拒绝或提出修改
- 没有得到确认的关系不能生效

### 3. 最小权限原则 🔐
**默认无权限，按需授权**

- 建立关系时必须明确指定权限
- 默认权限为空（无任何权限）
- 只授予完成任务所需的最小权限
- 权限可以随时降级或撤销

### 4. 可撤销原则 ↩️
**任何关系都可单方面终止**

- 关系的任何一方都可以随时撤销关系
- 撤销关系时应该通知对方
- 撤销后权限立即失效
- 撤销是合法的权利，不需要理由

### 5. 透明原则 📋
**所有关系操作都有审计记录**

- 关系的建立、修改、撤销都有日志
- 日志记录：谁、何时、对谁、做了什么
- 审计日志不可篡改
- 只有授权方可以查看相关日志

### 6. 隐私原则 🛡️
**关系详情仅对当事人可见**

- 一个节点的关系图谱不对外公开
- 只有关系中的节点可以看到关系详情
- 第三方无法知道 A 和 B 是否有关系
- 除非 A 或 B 主动披露

## 关系类型

### observe（观察）
- 用途：监护、监控、关注
- 典型场景：父母观察孩子、管理员监控服务
- 默认权限：read

### manage（管理）
- 用途：设备管理、系统管理
- 典型场景：管理 IoT 设备、管理服务器
- 默认权限：read, write

### collaborate（协作）
- 用途：平等协作、联合工作
- 典型场景：团队成员协作、AI 之间协作
- 默认权限：read, write

### delegate（委托）
- 用途：授权代理、任务委托
- 典型场景：委托 AI 代理、临时授权
- 默认权限：按需指定

---

## 权限级别

| 权限 | 说明 |
|------|------|
| `read` | 可以读取节点的公开信息 |
| `write` | 可以向节点发送消息/数据 |
| `admin` | 可以管理节点的配置/关系 |

**权限组合：**
- `read` alone：只读观察
- `read + write`：可以双向交互
- `read + write + admin`：完全控制

## 关系建立流程

```
步骤 1: 发起请求
┌─────────────────────────────────────────────┐
│ 节点 A → ClawNet                             │
│ "我想与 B 建立 observe 关系，请求 read 权限" │
└─────────────────────────────────────────────┘
                    ↓
步骤 2: ClawNet 验证
┌─────────────────────────────────────────────┐
│ ✓ A 存在且有效                               │
│ ✓ B 存在且有效                               │
│ ✓ A 有权发起请求                             │
│ ✓ 请求符合关系宪章                           │
└─────────────────────────────────────────────┘
                    ↓
步骤 3: 通知目标节点
┌─────────────────────────────────────────────┐
│ ClawNet → B                                 │
│ "A 想观察你，请求 read 权限，同意吗？"       │
│ [同意] [拒绝] [修改权限]                     │
└─────────────────────────────────────────────┘
                    ↓
步骤 4: 目标节点响应
┌─────────────────────────────────────────────┐
│ B → ClawNet                                 │
│ "同意，授予 read 权限"                       │
└─────────────────────────────────────────────┘
                    ↓
步骤 5: 关系建立
┌─────────────────────────────────────────────┐
│ ClawNet                                     │
│ ✅ 建立关系：A →(observe, read)→ B          │
│ 📝 记录审计日志                              │
│ 🔔 通知 A 关系已建立                         │
└─────────────────────────────────────────────┘
```

```
                    ↓
步骤 4: 记录审计
┌─────────────────────────────────────────────┐
│ ClawNet                                     │
│ 📝 记录撤销操作到审计日志                    │
└─────────────────────────────────────────────┘
```

## 安全措施

### 1. 身份验证
- 每个节点必须有唯一标识
- 关系操作需要身份验证
- 支持 Token、OAuth2、签名等多种认证方式

### 2. 权限隔离
- 不同关系的权限相互独立
- 权限不自动继承（除非明确配置）
- 权限检查在每次操作时执行

### 3. 防滥用
- 关系请求频率限制
- 拒绝率过高的节点会被标记
- 可设置黑名单/白名单

### 4. 数据保护
- 关系数据加密存储
- 传输使用 HTTPS/WSS
- 敏感字段脱敏处理
